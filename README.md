### 架构设计

| 技术点 | 实现方案 | 达成效果 |
|--------|----------|----------|
| **基于组件的模块化架构** | 采用Entity-Component模式拆分Player、Enemy、Item等核心实体，通过接口（`IInteractable`、`IDamageable`）定义交互契约 | 消除类间硬编码依赖，新增敌人类型无需修改Player代码，符合开闭原则 |
| **事件驱动的消息系统** | 实现泛型`EventBus<T>`单例，支持任意模块订阅/发布游戏事件（如OnPlayerDeath、OnLevelComplete） | 解耦系统间直接引用，UI层与Gameplay层零耦合通信，便于单元测试 |

## 🏗️ 项目架构设计

```
Assets/
├── _Project/                          # 项目核心代码（避免与Plugins冲突）
│   ├── Scripts/
│   │   ├── Core/                      # 框架层：基础设施与通用工具
│   │   │   ├── EventSystem/           # 事件总线实现
│   │   │   ├── ObjectPool/            # 对象池系统
│   │   │   ├── StateMachine/          # 状态机框架
│   │   │   └── Utilities/             # 扩展方法、数学工具等
│   │   ├── Gameplay/                  # 游戏逻辑层
│   │   │   ├── Player/                # 玩家角色：输入处理、状态机、技能
│   │   │   ├── Enemy/                 # 敌人AI：寻路、行为树、战斗逻辑
│   │   │   ├── Interactables/         # 可交互对象：道具、机关、NPC
│   │   │   └── Level/                 # 关卡管理：生成、进度、存档点
│   │   ├── Systems/                   # 全局系统层
│   │   │   ├── GameManager/           # 游戏生命周期、全局状态
│   │   │   ├── SaveLoad/              # 存档系统：JSON/Binary序列化
│   │   │   ├── Audio/                 # 音频管理：BGM、SFX、混音
│   │   │   └── SceneManagement/       # 场景加载、过渡动画
│   │   └── UI/                        # UI层
│   │       ├── Views/                 # 界面预制体与MonoBehaviour
│   │       ├── Presenters/            # UI逻辑与数据绑定
│   │       └── Widgets/               # 可复用UI组件
│   ├── Resources/                     # 运行时动态加载资源
│   │   ├── Prefabs/                   # 预制体分类存储
│   │   ├── ScriptableObjects/         # 配置数据：敌人数值、关卡参数
│   │   └── AddressableAssets/         # Addressables资源组
│   ├── Scenes/                        # 场景文件
│   │   ├── Boot/                      # 初始化场景
│   │   ├── MainMenu/                  # 主菜单
│   │   └── Levels/                    # 关卡场景
│   └── Settings/                      # 配置文件
│       ├── Input/                     # Input Action Assets
│       └── Render/                    # URP/HDRP配置
└── Plugins/                           # 第三方插件
    └── 【插件名称】/
```

### 关键设计决策

- **分层原则**：Core层零业务依赖，Gameplay层通过接口与Core通信，确保框架可复用
- **数据驱动**：敌人属性、关卡配置均采用ScriptableObject，支持策划快速调整平衡性
- **单一职责**：每个MonoBehaviour只负责一个功能（如`PlayerMovement`与`PlayerAnimation`分离）

## 📝 代码规范说明

- **命名规范**：类/方法采用PascalCase，字段采用_camelCase，常量采用UPPER_SNAKE_CASE
- **文件组织**：每个类单独文件，命名与类名完全一致；文件夹按功能模块划分而非按类型
- **注释标准**：公共API使用XML文档注释，复杂算法处附原理说明；避免无意义注释
- **Git规范**：采用语义化提交（feat:/fix:/refactor:/docs:），功能分支开发，PR需通过Code Review
